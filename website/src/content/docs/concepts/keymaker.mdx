Most neograph topologies are static: the edges are fixed when you compile. `Keymaker` adds a *mesh* -- a set of nodes where each member decides at RUNTIME which declared peer to hand off to. A triage node routes a ticket to the right specialist; a specialist can route BACK to triage for re-routing (a genuine cycle); a `max_hops` budget guarantees the mesh terminates. Routing is a runtime `Command(goto)`, not a conditional edge fixed at assembly -- yet every reachable peer is type-checked when you build the construct.

This page covers the SHIPPED surface: **mode (a) peer routing**. Mode (b), dynamic flow dispatch, is described at the end as a not-yet-runnable stub.

## The use case: a typed support swarm

A support ticket arrives. A `triage` node inspects it and hands off to one of several declared specialist peers -- billing, technical, account. A specialist either resolves the ticket (routes to the terminal sentinel) or, if it turns out not to own the ticket, hands it BACK to triage. A static conditional edge cannot express "route back to a node I already ran"; a mesh can, and the hop budget keeps the loop finite.

## Syntax

### @node decorator

The `peers=` keyword builds a `Keymaker` for you (the exact mirror of `loop_when=` building a `Loop`). `max_hops=` is an entry-only knob -- it belongs on the member that starts the mesh. Every member reads the previous payload through the reserved `handoff` input and returns the shared payload model, whose routing field (`goto` by default) names the next peer or `HANDOFF_END`.

```python
from typing import Annotated
from pydantic import BaseModel
from neograph import HANDOFF_END, FromInput, node, construct_from_functions, compile, run

class Handoff(BaseModel, frozen=True):
    goto: str                      # a declared peer name, or HANDOFF_END
    subject: str
    resolution: str | None = None
    hops: int = 0

@node(outputs=Handoff, peers=["billing"], max_hops=6)
def triage(handoff: Handoff, subject: Annotated[str, FromInput] = "") -> Handoff:
    if handoff is None:            # first activation: the mesh channel is empty
        return Handoff(goto="billing", subject=subject, hops=1)
    return Handoff(goto=HANDOFF_END, subject=handoff.subject, hops=handoff.hops + 1)

@node(outputs=Handoff, peers=[])   # a terminal specialist routes only to HANDOFF_END
def billing(handoff: Handoff) -> Handoff:
    return Handoff(goto=HANDOFF_END, subject=handoff.subject,
                   resolution="Refund issued.", hops=handoff.hops + 1)

mesh = construct_from_functions("swarm", [triage, billing])
graph = compile(mesh)
result = run(graph, input={"subject": "Double charge"})
print(result["billing"].resolution)   # -> Refund issued.
```

The reserved `handoff` parameter must be typed as the payload model exactly (not `Handoff | None`); at runtime the entry member receives `None` on its first activation because the shared channel is still empty.

### Programmatic (pipe) form

The same mesh, built declaratively -- `Node.scripted(...) | Keymaker(...)` produces IDENTICAL mesh IR and routing. The keyless bodies arrive through `compile(scripted=...)`.

```python
from neograph import Node, Keymaker, Construct, compile, run

def fn_triage(input_data, _config):
    incoming = input_data.get("handoff") if isinstance(input_data, dict) else None
    if incoming is None:
        return Handoff(goto="billing", subject="Double charge", hops=1)
    return Handoff(goto=HANDOFF_END, subject=incoming.subject, hops=incoming.hops + 1)

def fn_billing(input_data, _config):
    incoming = input_data["handoff"]
    return Handoff(goto=HANDOFF_END, subject=incoming.subject,
                   resolution="Refund issued.", hops=incoming.hops + 1)

mesh = Construct("swarm-declarative", nodes=[
    Node.scripted("triage", fn="fn_triage", outputs=Handoff) | Keymaker(peers=["billing"], max_hops=6),
    Node.scripted("billing", fn="fn_billing", inputs={"handoff": Handoff}, outputs=Handoff) | Keymaker(peers=[]),
])
graph = compile(mesh, scripted={"fn_triage": fn_triage, "fn_billing": fn_billing})
print(run(graph, input={})["billing"].resolution)   # -> Refund issued.
```

The `@node(peers=...)` sugar and the declarative pipe form are two of neograph's three surfaces. ForwardConstruct is exempt in v1: a runtime mesh has no static dataflow for the tracer to thread, so there is nothing for `forward()` to trace into a `Keymaker`.

## How it works

At compile time neograph lowers a `Keymaker` member into a wrapper that returns a LangGraph `Command(goto=..., update=...)`, so control flow is derived at runtime from the node's output:

1. **Routing field**: the wrapper reads `Keymaker.route` (default `"goto"`) off the member's payload. That value must be a declared peer or `HANDOFF_END` -- a target outside `peers` raises before any goto is issued, so a typo is a loud error, not a silent misroute.
2. **The shared channel**: every member reads the previous payload through the reserved `handoff` input. The channel is keyed off the entry member, so all peers read and write the same slot -- the framework fields are `neo_`-prefixed and never touch your schema.
3. **The hop budget**: `Keymaker.max_hops` bounds the cycle (the mesh analogue of `Loop`'s `max_iterations`). `HANDOFF_END` is a clean exit -- never counted against the budget.

### max_hops and on_exhaust

`Keymaker.max_hops` is declared on the entry member only. When the budget is spent, `Keymaker.on_exhaust` decides what happens:

- `"error"` (default) -- raise, surfacing the runaway mesh.
- `"exit"` -- leave the mesh cleanly with the last payload on the bus.

```python
from neograph import node, construct_from_functions, compile, run
from neograph.errors import ExecutionError

class Ping(BaseModel, frozen=True):
    goto: str
    hops: int = 0

@node(outputs=Ping, peers=["pong"], max_hops=3, on_exhaust="error")
def ping(handoff: Ping) -> Ping:
    return Ping(goto="pong", hops=(0 if handoff is None else handoff.hops + 1))

@node(outputs=Ping, peers=["ping"])
def pong(handoff: Ping) -> Ping:
    return Ping(goto="ping", hops=handoff.hops + 1)

graph = compile(construct_from_functions("pingpong", [ping, pong]))
try:
    run(graph, input={})
except ExecutionError as exc:
    print("budget stopped the loop")
```

Note the surface asymmetry: `Keymaker.on_exhaust` is `Literal['error', 'exit']`, but the `@node` decorator's `on_exhaust=` keyword accepts `'error' | 'last' | 'exit'` -- it is shared with `Loop` and routed by which trigger you used (`peers=` builds a `Keymaker`; `loop_when=` builds a `Loop`). Only `'error'` and `'exit'` are meaningful for a mesh.

## What the mesh guarantees -- and what it does not

neograph's linear validator normally proves "exactly one type-checked producer per consumer". Inside a mesh that property does not hold, and the docs should say so plainly. What `Keymaker` mode (a) guarantees STATICALLY is:

- every declared peer exists in the construct;
- every hop payload type-checks against every peer's `handoff` input (the mesh is uniform -- one payload model);
- the mesh terminates (the `max_hops` budget).

What it does NOT prove statically is WHICH member produced the payload a given peer receives -- that is runtime information, because the routing is a runtime `Command(goto)`. This is the same cross-arm limitation a branch has, generalized to a cycle. You trade a compile-time producer identity for runtime routing; the payload TYPE contract is still enforced everywhere.

## Durability

Each hop is one superstep, so with a checkpointer every handoff is its own checkpoint and the hop counter persists across resume. The mesh channel and hop-counter fields are `neo_`-prefixed and excluded from the schema fingerprint, so they never trigger a spurious checkpoint rewind.

### Human-in-the-loop around a mesh

An `Operator` (the human-in-the-loop gate) may NOT sit on a mesh member -- a member compiles to a `Command(goto)` wrapper with no room for an interrupt, and mixing the two is rejected at assembly. Express approval by placing the `Operator` BEFORE the mesh entry (gate the ticket before it is routed) or AFTER the mesh exit (review the resolution before it ships). The gate wraps the mesh; it does not live inside it.

## Dynamic flow dispatch (route='decide')

`Keymaker` is one modifier with two modes. Beside peer routing, it declares a second mode -- `Keymaker(route="decide", spec_field=..., input_field=..., output=..., scripted=...)` -- in which a member dispatches a whole sub-flow chosen at runtime from a spec. This mode **config-validates at assembly today** (the `Keymaker.spec_field` / `Keymaker.input_field` / `Keymaker.output` fields are accepted and checked), but it is **not yet dispatchable at runtime** -- there is no execution wrapper for it in this release. Treat it as a reserved surface: do not wire a `route="decide"` member into a pipeline you intend to run yet. This section will be replaced with runnable documentation when the dispatch path ships.
