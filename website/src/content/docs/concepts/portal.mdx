Most neograph topologies are static: the edges are fixed when you compile. `Portal` adds a *mesh* -- a set of nodes where each member decides at RUNTIME which declared peer to hand off to. A triage node routes a ticket to the right specialist; a specialist can route BACK to triage for re-routing (a genuine cycle); a `max_hops` budget guarantees the mesh terminates. Routing is a runtime `Command(goto)`, not a conditional edge fixed at assembly -- yet every reachable peer is type-checked when you build the construct.

`Portal` is one modifier with two modes. **Mode (a), peer routing** (most of this page) turns a fixed set of nodes into a mesh. **Mode (b), dynamic flow dispatch** (the last section) lets a member choose a WHOLE sub-flow at runtime from an emitted spec -- both are shipped and runnable.

## The use case: a typed support swarm

A support ticket arrives. A `triage` node inspects it and hands off to one of several declared specialist peers -- billing, technical, account. A specialist either resolves the ticket (routes to the terminal sentinel) or, if it turns out not to own the ticket, hands it BACK to triage. A static conditional edge cannot express "route back to a node I already ran"; a mesh can, and the hop budget keeps the loop finite.

## Syntax

### @node decorator

The `portal=` keyword builds a `Portal` for you (the exact mirror of `loop_when=` building a `Loop`). `max_hops=` is an entry-only knob -- it belongs on the member that starts the mesh. Every member reads the previous payload through the reserved `handoff` input and returns the shared payload model, whose routing field (`goto` by default) names the next peer or `HANDOFF_END`.

```python
from typing import Annotated
from pydantic import BaseModel
from neograph import HANDOFF_END, FromInput, node, construct_from_functions, compile, run

class Handoff(BaseModel, frozen=True):
    goto: str                      # a declared peer name, or HANDOFF_END
    subject: str
    resolution: str | None = None
    hops: int = 0

@node(outputs=Handoff, portal=["billing"], max_hops=6)
def triage(handoff: Handoff, subject: Annotated[str, FromInput] = "") -> Handoff:
    if handoff is None:            # first activation: the mesh channel is empty
        return Handoff(goto="billing", subject=subject, hops=1)
    return Handoff(goto=HANDOFF_END, subject=handoff.subject, hops=handoff.hops + 1)

@node(outputs=Handoff, portal=[])   # a terminal specialist routes only to HANDOFF_END
def billing(handoff: Handoff) -> Handoff:
    return Handoff(goto=HANDOFF_END, subject=handoff.subject,
                   resolution="Refund issued.", hops=handoff.hops + 1)

mesh = construct_from_functions("swarm", [triage, billing])
graph = compile(mesh)
result = run(graph, input={"subject": "Double charge"})
print(result["billing"].resolution)   # -> Refund issued.
```

The reserved `handoff` parameter must be typed as the payload model exactly (not `Handoff | None`); at runtime the entry member receives `None` on its first activation because the shared channel is still empty.

### Programmatic (pipe) form

The same mesh, built declaratively -- `Node.scripted(...) | Portal(...)` produces IDENTICAL mesh IR and routing. The keyless bodies arrive through `compile(scripted=...)`.

```python
from neograph import Node, Portal, Construct, compile, run

def fn_triage(input_data, _config):
    incoming = input_data.get("handoff") if isinstance(input_data, dict) else None
    if incoming is None:
        return Handoff(goto="billing", subject="Double charge", hops=1)
    return Handoff(goto=HANDOFF_END, subject=incoming.subject, hops=incoming.hops + 1)

def fn_billing(input_data, _config):
    incoming = input_data["handoff"]
    return Handoff(goto=HANDOFF_END, subject=incoming.subject,
                   resolution="Refund issued.", hops=incoming.hops + 1)

mesh = Construct("swarm-declarative", nodes=[
    Node.scripted("triage", fn="fn_triage", outputs=Handoff) | Portal(to=["billing"], max_hops=6),
    Node.scripted("billing", fn="fn_billing", inputs={"handoff": Handoff}, outputs=Handoff) | Portal(to=[]),
])
graph = compile(mesh, scripted={"fn_triage": fn_triage, "fn_billing": fn_billing})
print(run(graph, input={})["billing"].resolution)   # -> Refund issued.
```

The `@node(portal=...)` sugar and the declarative pipe form are two of neograph's three surfaces. ForwardConstruct is exempt in v1: a runtime mesh has no static dataflow for the tracer to thread, so there is nothing for `forward()` to trace into a `Portal`.

## How it works

At compile time neograph lowers a `Portal` member into a wrapper that returns a LangGraph `Command(goto=..., update=...)`, so control flow is derived at runtime from the node's output:

1. **Routing field**: the wrapper reads `Portal.route` (default `"goto"`) off the member's payload. That value must be a declared peer or `HANDOFF_END` -- a target outside `peers` raises before any goto is issued, so a typo is a loud error, not a silent misroute.
2. **The shared channel**: every member reads the previous payload through the reserved `handoff` input. The channel is keyed off the entry member, so all peers read and write the same slot -- the framework fields are `neo_`-prefixed and never touch your schema.
3. **The hop budget**: `Portal.max_hops` bounds the cycle (the mesh analogue of `Loop`'s `max_iterations`). `HANDOFF_END` is a clean exit -- never counted against the budget.

### max_hops and on_exhaust

`Portal.max_hops` is declared on the entry member only. When the budget is spent, `Portal.on_exhaust` decides what happens:

- `"error"` (default) -- raise, surfacing the runaway mesh.
- `"exit"` -- leave the mesh cleanly with the last payload on the bus.

```python
from neograph import node, construct_from_functions, compile, run
from neograph.errors import ExecutionError

class Ping(BaseModel, frozen=True):
    goto: str
    hops: int = 0

@node(outputs=Ping, portal=["pong"], max_hops=3, on_exhaust="error")
def ping(handoff: Ping) -> Ping:
    return Ping(goto="pong", hops=(0 if handoff is None else handoff.hops + 1))

@node(outputs=Ping, portal=["ping"])
def pong(handoff: Ping) -> Ping:
    return Ping(goto="ping", hops=handoff.hops + 1)

graph = compile(construct_from_functions("pingpong", [ping, pong]))
try:
    run(graph, input={})
except ExecutionError as exc:
    print("budget stopped the loop")
```

Note the surface asymmetry: `Portal.on_exhaust` is `Literal['error', 'exit']`, but the `@node` decorator's `on_exhaust=` keyword accepts `'error' | 'last' | 'exit'` -- it is shared with `Loop` and routed by which trigger you used (`portal=` builds a `Portal`; `loop_when=` builds a `Loop`). Only `'error'` and `'exit'` are meaningful for a mesh.

## What the mesh guarantees -- and what it does not

neograph's linear validator normally proves "exactly one type-checked producer per consumer". Inside a mesh that property does not hold, and the docs should say so plainly. What `Portal` mode (a) guarantees STATICALLY is:

- every declared peer exists in the construct;
- every hop payload type-checks against every peer's `handoff` input (the mesh is uniform -- one payload model);
- the mesh terminates (the `max_hops` budget).

What it does NOT prove statically is WHICH member produced the payload a given peer receives -- that is runtime information, because the routing is a runtime `Command(goto)`. This is the same cross-arm limitation a branch has, generalized to a cycle. You trade a compile-time producer identity for runtime routing; the payload TYPE contract is still enforced everywhere.

## Durability

Each hop is one superstep, so with a checkpointer every handoff is its own checkpoint and the hop counter persists across resume. The mesh channel and hop-counter fields are `neo_`-prefixed and excluded from the schema fingerprint, so they never trigger a spurious checkpoint rewind.

### Human-in-the-loop around a mesh

An `Operator` (the human-in-the-loop gate) may NOT sit on a mesh member -- a member compiles to a `Command(goto)` wrapper with no room for an interrupt, and mixing the two is rejected at assembly. Express approval by placing the `Operator` BEFORE the mesh entry (gate the ticket before it is routed) or AFTER the mesh exit (review the resolution before it ships). The gate wraps the mesh; it does not live inside it.

## Dynamic flow dispatch (route='decide')

Peer routing chooses the next NODE at runtime. The second mode goes further: it chooses a whole SUB-FLOW at runtime. A `Portal(route="decide", ...)` member -- the "dispatcher" -- emits, as its own typed output, a neograph-flavored *Agent Spec* dict (the SAME format `to_agent_spec()` exports / `from_agent_spec()` imports) plus the input that flow should run on. The framework then deserializes, validates, compiles, and runs that emitted flow, and writes its typed result back onto the bus for the rest of the pipeline. The topology did not exist until the dispatcher produced it.

This is the runtime-assembly path (see [LLM-driven pipelines](/runtime/llm-driven/)) turned into a first-class construct: instead of your application glue calling `from_agent_spec` + `compile` + `run` by hand, a `route="decide"` node does it inside the graph, under the same validation and observability as every other node.

### The surface

```python
from pydantic import BaseModel
from neograph import Construct, Portal, Node, compile, run, to_agent_spec

class Decision(BaseModel):      # the dispatcher's OWN output
    spec: dict                  # a neograph-flavored Agent Spec dict -- the flow to run
    flow_input: dict            # the input that flow runs on

class Summary(BaseModel, frozen=True):
    text: str

# A pre-registered building block the emitted flow may wire (see the boundary below).
def summarize(_input_data, _config) -> Summary:
    return Summary(text="done")

def plan(_input_data, config) -> Decision:
    # A real planner builds the spec dict by exporting a Construct through
    # to_agent_spec() -- the SAME canonical emitter, never a hand-authored
    # envelope (so the emitted spec stays honest to what from_agent_spec expects).
    chosen = Construct("chosen-flow", nodes=[Node.scripted("s", fn="summarize", outputs=Summary)])
    spec = to_agent_spec(chosen).to_dict()
    return Decision(spec=spec, flow_input={})

km = Portal(
    route="decide",
    spec_field="spec",          # which field on the payload holds the spec dict
    input_field="flow_input",   # which field holds the flow's input
    output=Summary,             # the type the dispatched flow must produce
    max_depth=5,                 # self-extending-flow recursion budget (required, no default)
    scripted={"summarize": summarize},   # the ONLY blocks the emitted flow may wire
)
planner = Node.scripted("planner", fn="plan", outputs=Decision) | km

graph = compile(Construct("dispatch", nodes=[planner]), scripted={"plan": plan})
result = run(graph, input={})
print(result["planner_dispatch"].text)   # -> done
```

The dispatched flow's result lands on a regular, fingerprinted state field named `{node}_dispatch` (here `planner_dispatch`). A downstream node consumes it like any other upstream: `Node.scripted("consumer", inputs={"planner_dispatch": Summary}, ...)`. See `examples/29_portal_dynamic_flow.py` for a planner that emits two different topologies (a one-node and a two-node flow) depending on its runtime input.

### The contract: if it dispatches, it was validated

The safety property is the whole reason this is a construct and not a raw `eval`. When the dispatcher fires, the framework runs four steps before any dispatched node executes:

1. **`AgentSpecDeserializer().from_dict(spec)` -> `from_agent_spec(flow)`** -- the emitted dict is deserialized and imported into a `Construct`, which runs the SAME `_validate_node_chain` gate a hand-written pipeline passes through. No bespoke validator, no relaxed schema. A machine-emitted spec is held to the identical standard as human-authored source.
2. **Output-contract check** -- if the built flow declares an output boundary, it must equal `Portal.output`.
3. **`compile(...)`** with the pre-registered `scripted=` / `conditions=` blocks.
4. **Invoke** the compiled flow on `input_field`'s data, extract the value assignable to `Portal.output`, and write it to `{node}_dispatch`.

So "if it dispatches, it was validated" holds even for a flow that did not exist until runtime -- the durability and type promises are not weakened by dynamic assembly.

### The capability boundary: composition, not code injection

An emitted flow may wire ONLY the building blocks you pre-registered on the `Portal` (`scripted={...}`, and `conditions={...}` for edges). The spec references them by string name; an unknown `scripted_fn` fails loud at compile. A machine-authored flow therefore COMPOSES pre-approved code -- it can select and connect your blocks in new topologies, but it can never introduce new executable code. This is the line that makes runtime dispatch safe to hand to an LLM planner.

### Rejection semantics

An invalid emitted spec is rejected at step 1, and the failure is loud and early. The underlying error (say, an agent/act-mode node -- `to_agent_spec` can lower it, but `from_agent_spec`'s import-side reconstruction is a tracked follow-up) surfaces WRAPPED in an `ExecutionError` that names the offending spec -- raised BEFORE any dispatched sub-node body runs. Demo 2 of the example proves this with a sentinel: the invalid flow's body never executes, because validation happens before execution.

### Durability caveat

In v1 the dispatched inner flow compiles with NO checkpointer. On a parent resume, the whole `route="decide"` node re-executes (re-emit the spec, re-validate, re-run) rather than resuming the inner flow from a mid-flow checkpoint. This is documented-opaque: the inner flow is not independently durable yet. Do not rely on inner-flow checkpoint resume; deeper per-dispatch durability is tracked separately (Tier-2). Everything OUTSIDE the dispatch node checkpoints normally.

### Three-surface note

Mode (b) is built through the declarative pipe form (`Node.scripted(...) | Portal(route="decide", ...)`) or the equivalent `Node(name=..., mode="scripted", ...) | Portal(...)`. There is no `@node` decorator sugar for it in v1: a runtime-emitted flow has no static dataflow for the decorator (or ForwardConstruct) to build, so there is nothing to thread -- the same exemption peer routing gives ForwardConstruct.
